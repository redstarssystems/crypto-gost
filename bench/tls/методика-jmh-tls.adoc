= Методика измерений скорости работы ГОСТ TLS 1.3 crypto-gost-tls13
:toc:
:toc-title: Содержание
:source-highlighter: rouge
:icons: font

== Введение

Цель — получить замеры насколько быстро работает ГОСТ TLS 1.3. +

.Мы измеряем несколько вещей:
* полное установление защищённого соединения (handshake — «рукопожатие», обмен приветствиями между клиентом и сервером);
* ускоренное возобновление соединения через заранее согласованный ключ (PSK — pre-shared key);
* отдельные криптографические операции внутри рукопожатия;
* скорость шифрования и расшифровки уже установленного соединения (record layer);
* скорость хеширования Streebog-256 на данных разного размера.

Каждый набор тестов отвечает на следующие вопросы:

[cols="1,3"]
|===
| Набор тестов | Вопрос

| `FullHandshakeBench`
| Сколько полных рукопожатий TLS 1.3 одно ядро процессора успевает сделать за секунду?

| `PskResumptionBench`
| Во сколько раз быстрее идёт ускоренное рукопожатие через PSK по сравнению с полным?

| `HandshakeComponentsBench`
| Что именно тормозит рукопожатие: обмен ключами (ECDHE), создание подписи, проверка подписи или вывод ключей?

| `RecordProtectBench`
| Сколько данных в секунду пропускает уровень записей (шифрование) после того, как соединение уже установлено?

| `Streebog256Bench`
| Как меняется скорость хеширования Streebog-256 в зависимости от размера входных данных?
|===

== Инструменты

[cols="1,3"]
|===
| Инструмент измерения | JMH версии 1.37 (стандартный инструмент для замеров скорости Java-кода)
| Версия Java | 21 и новее
| Как запускается | Собирается один большой исполняемый JAR-файл, запускается командой `java -jar benchmarks.jar`
| Что измеряется | Модуль `crypto-gost-tls13` (и вместе с ним `crypto-gost-core`)
|===

Важная деталь: тесты *не используют настоящую сеть*. +
Все замеры рукопожатия идут через `InMemoryTlsTransport` — канал связи внутри памяти компьютера. Так мы убираем влияние сети из результатов и измеряем только то, что нас интересует: чистую работу криптографии и протокола.

== Общие настройки замеров

Все тесты используют одинаковый набор настроек:

[source,java]
----
@BenchmarkMode(Mode.Throughput)        // измеряем «операций в секунду»
@OutputTimeUnit(TimeUnit.SECONDS)      // единица времени — секунда
@State(Scope.Benchmark)                // общее состояние для всех замеров
----

*Почему измеряем «операций в секунду», а не «время одной операции»:*

* главная задача замеров — планирование мощности _capasity planning_ (сколько серверов нужно под нагрузку). Для этого важнее знать «сколько помещается в секунду», чем «сколько длится одна операция»;
* JMH  выдаёт обе цифры, но «операций в секунду» для нас основная.

*Почему состояние общее, а не отдельное для каждого потока:*

* все тесты работают в один поток;
* подготовку, в виде тяжелых операций (создание ключей, сертификатов, PSK-билетов) делаем один раз и потом переиспользуем.

== Этапы подготовки и завершения

JMH различает несколько уровней подготовки и очистки:

[cols="1,3"]
|===
| Уровень «весь тест» (`Level.Trial`)
| Выполняется один раз — до начала всех замеров и после их окончания. Здесь делается вся подготовка: создание серверных ключей и сертификата, запуск пула потоков, одно «эталонное» рукопожатие для получения PSK-билета.

| Уровень «одна итерация» (`Level.Iteration`)
| В наших тестах не используется.

| Уровень «один вызов» (`Level.Invocation`)
| Выполняется перед и после каждого вызова измеряемого метода. Намеренно избегаем: наши операции длятся десятки микросекунд — миллисекунды, и затраты на саму подготовку на этом уровне стали бы сравнимы с измеряемой операцией — то есть исказили бы результат.
|===

Единственное исключение — сброс счётчика номера записи (sequence number) в тесте `RecordProtectBench.unprotect`. Там одна простая запись числа (примерно 1 наносекунда) делается прямо внутри измеряемого метода. Эта погрешность ничтожна по сравнению с самой операцией расшифровки, которая занимает микросекунды.

== Прогон на разных размерах данных

Два теста прогоняются на нескольких размерах входных данных:

[source,java]
----
// RecordProtectBench
@Param({"100", "1024", "16383"})  // байт в одной TLS-записи

// Streebog256Bench
@Param({"32", "1024", "16384"})   // байт на вход Streebog-256
----

Размеры выбраны не случайно:

* *100 и 32 байта* — маленькое сообщение. Здесь основное время уходит на фиксированные накладные расходы (служебные данные шифрования: nonce и тег для шифрования; запуск и завершение для хеширования);
* *1024 байта* — типичный размер для HTTP-подзапросов;
* *16383 и 16384 байта* — предельный размер TLS-записи (без служебного заголовка).

Прогон на разных размерах используется *только в специализированных тестах*. Тест `HandshakeComponentsBench` его не использует — это было бы смешением несвязанных вещей: размер входных данных для Streebog не должен влиять на цифры создания ключей ECDHE или подписи.

== Как организована работа в несколько потоков

Тесты `PskResumptionBench` и `FullHandshakeBench` требуют, чтобы клиент и сервер работали одновременно (потому что канал в памяти — блокирующий: одна сторона ждёт другую).

Решение: один общий пул потоков на весь тест, серверное рукопожатие запускается асинхронно.

[source,java]
----
@Setup(Level.Trial)
public void setup() {
    exec = Executors.newVirtualThreadPerTaskExecutor();
    // ...
}

@TearDown(Level.Trial)
public void tearDown() {
    exec.shutdown();
}

@Benchmark
public TlsSession handshake() {
    // ...
    CompletableFuture<Void> serverFuture = CompletableFuture.runAsync(() -> {
        server.handshakeAsServer();
    }, exec);
    client.handshakeAsClient();   // главный поток
    serverFuture.join();          // ждём сервер ПОСЛЕ рукопожатия клиента
    // ...
}
----

Что здесь критически важно:

* *создавать новый поток на каждой итерации было бы катастрофой*: при примерно 40 операциях в секунду это 80 операций создания, запуска и остановки потоков в секунду — тогда замер измерял бы скорость работы с потоками, а не скорость рукопожатия;
* *запуск асинхронной задачи без указания конкретного пула потоков* использовал бы общий пул на всю программу — а в нём плавающая нагрузка от других тестов давала бы шумные, нестабильные цифры;
* *виртуальные потоки* (`newVirtualThreadPerTaskExecutor`) выбраны потому, что: (а) проект в целом ориентирован на виртуальные потоки; (б) они дёшевы в создании и уничтожении; (в) корректно работают с блокирующим каналом в памяти.

Порядок операций критичен: сначала `runAsync(server)` запускает серверное рукопожатие в виртуальном потоке, затем главный поток делает рукопожатие клиента, и только после клиента `serverFuture.join()` дожидается сервер. Обратный порядок (ожидание сервера до рукопожатия клиента) привёл бы к взаимной блокировке: сервер застрял бы в ожидании приветствия от клиента, которое ещё не отправлено.

== Насколько реалистичны входные данные

=== Ключи шифрования в RecordProtectBench

В тесте используются *случайные* ключи, а не ключи, полученные из настоящего рукопожатия.

.Это осознанное решение:
* скорость шифрования Кузнечик-MGM не зависит от того, как получены ключи — она зависит только от длины ключа и размера данных. Случайные ключи функционально эквивалентны ключам из рукопожатия;
* это упрощает подготовку: не нужно прогонять целое рукопожатие перед замером уровня записей.

Этот момент отдельно прокомментирован в коде, чтобы разработчик не подумал, что это недоработка.

=== Подпись в HandshakeComponentsBench.verify

Подпись для теста проверки подписи вычисляется один раз на этапе подготовки:

[source,java]
----
@Setup(Level.Trial)
public void setup() {
    // ...
    precomputedSig = Signature.signHash(hash32, signKey);
}

@Benchmark
public boolean verify() {
    return Signature.verifyHash(hash32, precomputedSig, verifyKey);
}
----

Иначе тест проверки измерял бы «создание подписи плюс проверка» и завышал бы скорость проверки вдвое.

=== PSK-билет в PskResumptionBench

На этапе подготовки выполняется одно полное рукопожатие с явно настроенным хранилищем PSK на обеих сторонах. Серверное хранилище сохраняется как поле класса и переиспользуется в замере возобновления соединения. Без этого сервер не нашёл бы PSK-идентификатор и сделал бы полное рукопожатие вместо ускоренного.

[source,java]
----
private PskStore serverPskStore;  // живёт до конца всего теста

@Setup(Level.Trial)
public void setup() {
    serverPskStore = new InMemoryPskStore(100);
    // ... первое рукопожатие, серверное хранилище запоминает билет
}

@Benchmark
public TlsSession resume() {
    TlsSession server = TlsSession.createServer(...);
    server.setPskStore(serverPskStore);  // то же хранилище с билетом
    // ...
}
----

== Что входит в измеряемое время

[cols="1,2,2"]
|===
| Тест | Входит в замер | Не входит (сделано заранее)

| `FullHandshakeBench.handshake`
| Создание канала в памяти, двух TLS-сессий, полное рукопожатие, закрытие
| Создание сертификата, создание серверных ключей

| `PskResumptionBench.resume`
| То же плюс поиск PSK на сервере
| Первое рукопожатие (получение билета) — сделано на этапе подготовки

| `HandshakeComponentsBench.*`
| Одна криптографическая операция (создание ключей ECDHE, подпись, проверка, вывод ключей и т.п.)
| Установка ключей, вычисление хеша — на этапе подготовки

| `RecordProtectBench.protect`
| Один вызов шифрования — шифрование плюс вычисление тега
| Создание записи, вывод ключей — на этапе подготовки.

| `RecordProtectBench.unprotect`
| Сброс счётчика записи (примерно 1 наносекунда) плюс один вызов расшифровки — расшифровка плюс проверка тега
| То же

| `Streebog256Bench.hash`
| Создание экземпляра Streebog256, добавление данных, получение результата.
| Заполнение входных данных случайными байтами — на этапе подготовки.
|===

== Что мы намеренно не измеряем

* *Разбор сертификатов X.509, разбор ASN.1, разбор TLS-записей.* Если общие тесты (`FullHandshakeBench`) покажут проблему — точечно измерим по результатам профилирования. Измерять заранее — это преждевременная оптимизация.
* *Попадание и промах кеша OCSP.* Обращение раз в час, в 99% запросов данные уже в кеше. На скорость рукопожатия не влияет.
* *Поведение сборщика мусора.* Эти данные получаются бесплатно через флаг `-prof gc` к любому тесту, отдельный тест не нужен.
* *Проверка сертификатов.* Однократная операция на одно рукопожатие, уже учтена внутри `FullHandshakeBench`.

== Как запускать

Тесты запускаются командой:

[source,bash]
----
java -jar target/benchmarks.jar [фильтр] [флаги]
----

=== Быстрая проверка работоспособности

[source,bash]
----
java -jar target/benchmarks.jar -wi 1 -i 1 -f 1
----

* `-wi 1` — 1 разогревочная итерация (10 секунд)
* `-i 1` — 1 измерительная итерация (10 секунд)
* `-f 1` — 1 запуск в отдельной JVM

Полный прогон занимает примерно 10-15 минут. Цифры с погрешностью ±20-30%, *публиковать нельзя*. Этот режим нужен только для:

* проверки, что тесты не падают и не зависают;
* быстрой проверки на отсутствие регрессий после изменений в основном коде.

=== Боевой режим (для публикуемых цифр)

[source,bash]
----
java -jar target/benchmarks.jar -wi 5 -i 5 -f 3
----

* `-wi 5` — 5 разогревочных итераций (чтобы компилятор Java успел оптимизировать код).
* `-i 5` — 5 измерительных итераций (для усреднения).
* `-f 3` — 3 отдельных запуска JVM (чтобы исключить случайные особенности конкретного запуска).

Полный прогон занимает примерно 30-60 минут. Цифры с погрешностью ±2-5%. Инструмент JMH выдаёт колонку «погрешность результата» (`Score error`) — её обязательно включать в итоговые таблицы.

=== Запуск отдельного теста

[source,bash]
----
# Все тесты одного класса
java -jar target/benchmarks.jar FullHandshakeBench

# Один конкретный метод
java -jar target/benchmarks.jar RecordProtectBench.protect

# С анализом работы сборщика мусора
java -jar target/benchmarks.jar HandshakeComponentsBench -prof gc
----

== Что мы не делаем при запуске

* *Привязку к конкретному ядру процессора* (`taskset -c N`) — не используем. Инструмент JMH сам повышает приоритет процесса и изолирует запуски. Привязка к ядру имеет смысл только для микро-замеров с разбросом меньше 1%; у нас разброс — десятки процентов из-за ручного цикла рукопожатия, и привязка к ядру не даст полезного эффекта.
* *Анализ ассемблерного кода* (`-prof perfasm`) — по умолчанию не используем. Он полезен только при точечном исследовании узкого места — после того, как замер скорости его уже выявил.

== Воспроизводимость результатов

В каждом отчёте о замерах обязательно указывать:

* версию и сборку Java (`java -version`);
* версию JMH (1.37 — зафиксировано в настройках сборки);
* флаги запуска (`-wi N -i M -f K`);
* железо: модель процессора, число ядер, объём оперативной памяти;
* операционную систему и версию ядра (необязательно, но желательно — для понимания поведения, специфичного для Linux).

Без этих данных цифры из разных прогонов нельзя сравнивать между собой.

== Проверка результатов на адекватность

После каждого прогона стоит сверить результаты с ожиданиями:

[cols="1,2"]
|===
| Что сравниваем | Чего ожидаем

| Проверка подписи против создания подписи
| Должны различаться в разы — это норма для ГОСТ Р 34.10. Если совпадают — значит, либо создание подписи случайно попало в подготовку, либо проверка снова включает в себя создание.

| Вывод ключей (HKDF) против создания ключей ECDHE
| Вывод ключей должен быть в сотни-тысячи раз быстрее создания ключей ECDHE — это норма. Если скорости сравнимы — ошибка в создании ключей.

| Шифрование против расшифровки на одинаковом размере данных
| Должны быть примерно равны (расшифровка чуть быстрее за счёт проверки тега за один проход). Расхождение в 2 и более раза — ошибка.

| Streebog-256 на 32 байтах против 16384 байт (в мегабайтах в секунду)
| На 16 КБ скорость в мегабайтах в секунду должна быть выше, чем на 32 байтах (фиксированные расходы на запуск и завершение размазываются по большему объёму). Если скорость одинаковая — что-то не так с алгоритмом, нужно профилировать.

| Возобновление через PSK против полного рукопожатия
| Соотношение зависит от того, какую долю рукопожатия занимает обмен ключами ECDHE. В нашей реализации обмен ключами ECDHE остаётся даже в режиме возобновления (`psk_dhe_ke`), поэтому ускорение скромное — единицы или десятки процентов. Чтобы убедиться, что PSK действительно сработал, а не произошёл откат к полному рукопожатию, нужна явная проверка в коде теста (в текущей версии её нет, отложено на следующий этап).
|===
