== ADR 01

*ADR:* BigInteger arithmetic для вычисления s в ГОСТ Р 34.10-2012. +
*Контекст:* финальное вычисление подписи s = (r·d + k·e) mod n выполняется через java.math.BigInteger. +
*Решение:* оставить BigInteger, не реализовывать CT-арифметику по модулю n. +

*Обоснование:*

** Критическая операция k·G защищена лестницей Монтгомери — утечка k через скалярное умножение закрыта.
** Умножение фиксированных операндов (256/512 бит по параметрам кривой) — length-dependent ветвления Karatsuba не создают timing-вариации. Шаг mod() через divideKnuth содержит ветвления по промежуточным значениям частного и CT-гарантии не даёт.
** k детерминирован (RFC 6979) — статистические атаки на timing k·e требуют нереалистичного числа наблюдений в условиях JIT-компиляции, GC-пауз и планировщика потоков.
** CT-арифметика по n в JVM требует собственного арифметического ядра — ни одно из известных мне решений, ни стандартные JCA-провайдеры этого не делают.
** dBytes хранится как byte[] и обнуляется в destroy() — heap-lifetime закрытого ключа минимизирован там, где это возможно.

*Принятый риск:* теоретическая timing-уязвимость на уровне
BigInteger.mod(). В модели remote attacker — неизмерима: сетевой
jitter на порядки превышает timing-сигнал (~1 ns). Severity: LOW.

*Граница применимости:* данное решение (ADR) сделано для модели угроз
remote attacker — когда противник действует по сети,  без доступа к хосту. Это типичный сценарий для TLS-соединения. Локальный атакующий (co-located attacker) с произвольным кодом на том же хосте имеет более практичные пути компрометации ключа, чем
извлечение d через timing BigInteger.mod(). Требования к измерениям
(`10⁶`+ подписей в контролируемых условиях, сигнал в `20×` ниже шума
L3-измерений) делают эту атаку нереалистичной в production для любого
сценария. У локального атакующего есть более простые альтернативы.

*Пересмотреть если:* появится требование сертификации с явным требованием CT-арифметики, или выйдет практическая атака на BigInteger ECDSA s-value в JVM.
