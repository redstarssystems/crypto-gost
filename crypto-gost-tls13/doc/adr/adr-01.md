# ADR 01

**ADR:** certificate\_authorities extension не отправляется в
CertificateRequest (TLS 1.3).  
**Контекст:** при mTLS сервер отправляет CertificateRequest с
signature\_algorithms, но не включает certificate\_authorities (RFC 8446
§4.2.4). Клиент не получает список доверенных CA и не может
автоматически выбрать сертификат при нескольких подходящих.  
**Решение:** не включать certificate\_authorities в
CertificateRequest.  

**Обоснование:**

- Размер CertificateRequest фиксирован (23 байта) и укладывается в один
  TCP-сегмент. Список DN даже для 2–3 CA удлиняет сообщение на 100+ байт
  без пользы для ГОСТ-клиента.

- Раскрытие списка DN доверенных CA — информация для fingerprinting
  CA-инфраструктуры сервера. Защита от утечки метаданных согласуется с
  принципом наименьшего раскрытия. Валидация клиентской цепочки
  выполняется на сервере независимо от DN.

- Сервер хранит только публичные ключи CA (`List<PublicKeyParameters>`),
  а не полные X.509-сертификаты. Distinguished Names в DER-кодировке
  недоступны без загрузки сертификатов. Валидация цепочки идёт через
  `caPublicKey` напрямую, не через `acceptedIssuers` — список DN не
  участвует в проверке подписи.

**Принятый риск:** клиент с несколькими сертификатами от разных CA не
может автоматически выбрать подходящий сертификат. Необходимо
использовать конфигурацию `TlsClientConfig.withClientCertificateChain()`
или ручной выбор.

**Пересмотреть если:** появится поддержка нескольких CA с различными DN
в одном handshake, или клиентский API потребует автоматического выбора
сертификата по списку DN из CertificateRequest.
