# Назначение

Проверка устойчивости защищённого соединения TlsSession поверх
настоящего сокета под длительной смешанной нагрузкой. Тест отвечает на
вопросы, которые не покрывает замер пропускной способности: держит ли
соединение многих одновременных клиентов, переживает ли обрывы, не
накапливает ли память со временем, сколько соединений вмещает в память.

Тест: `TlsSessionStressTest` (пакет `org.rssys.gost.tls13.stress`,
состав crypto-gost-tls13). Состоит из двух самостоятельных проверок:
смешанная нагрузка (`stressTest`) и поиск предела по числу соединений
(`maxConcurrentSessions`).

ВАЖНО: это проверка устойчивости, а не скорости. Числа потока данных
(переданные мегабайты) здесь служебные и не являются мерой
производительности — для неё отдельный замер `TlsSessionStreamTest`.

# Проверка отсутствия утечки памяти

Каждые 10 секунд снимается объём занятой кучи. Решение принимается не по
среднему (оно искажается работой сборщика мусора), а по **нижней
границе** — наименьшему значению кучи в окне, то есть объёму памяти,
который сборщик не смог освободить. Нижняя граница отражает
действительно удерживаемые данные.

Порядок:

1.  Все замеры делятся на шесть равных по времени окон.

2.  Для каждого окна берётся наименьшее значение (нижняя граница).

3.  Вердикт выносится по второй половине теста — окнам с четвёртого по
    шестое. Первое окно (непрогретая среда JVM) и окна разогрева
    сборщика в расчёт не берутся.

Толкование:

- нижняя граница в окнах 4–6 держится в одном диапазоне (колеблется, но
  хотя бы в одном окне опускается обратно к прежним низким значениям) —
  **утечки нет**, это обычная работа сборщика GC;

- нижняя граница растёт от окна к окну (монотонно, то есть в каждом
  следующем окне не ниже предыдущего) **и** в шестом окне превышает
  четвёртое более чем в 1,2 раза — **подозрение на утечку**.

Если хотя бы одно из двух условий не выполнено (например, в каком-то
окне граница снова опустилась), это колебание сборщика, а не накопление.

Замечание: критерий рассчитан на заметное накопление. Очень медленная
утечка памяти с мелкими колебаниями нижней границы может не сработать ни
по монотонности, ни по порогу — её выявляют длительным наблюдением за
работающим сервером.

ВАЖНО: Длительность определяет, какую утечку тест способен увидеть.
Тридцать минут под полной нагрузкой исключают быстрое и среднее
накопление. Очень медленное накопление (на масштабе часов и суток) этим
тестом не выявляется. Такое нужно подтверждать наблюдением за работающим
сервером в реальной эксплуатации.

# Проверка 2 — предел по числу соединений

Соединения открываются и удерживаются группами с заданным шагом, пока
занятая куча не достигнет порога (например, 85 % от выделенной).
Достигнутое число — предельное количество одновременных соединений.

Различение причины остановки:

- Остановка при высокой занятости кучи — достигнут предел памяти
  (искомая величина);

- Частые ошибки соединений при низкой занятости кучи — остановка по
  ошибкам, не по памяти; результат помечается как недостоверный.

## Память на одно соединение

Перед замером удерживаемой памяти принудительно запускается сборка
мусора с проверкой стабилизации: сборка повторяется, пока изменение
занятой кучи между попытками не станет меньше 2% (но не более пяти
попыток). Без этого в величину затекает несобранный мусор и завышает её.

Память на соединение вычисляется как: **(удерживаемая куча − куча до
начала проверки) ÷ число соединений**.

ВАЖНО: это оценка сверху. В неё входит не только память соединений, но и
рост служебных данных среды исполнения под нагрузкой. Действительная
стоимость одного соединения ниже. Для расчёта ёмкости такая
консервативная оценка предпочтительна — она не завышает capacity.

# Что считается успехом

1.  Ноль ошибок по всем четырём видам нагрузки, включая обрывы.

2.  Нижняя граница Heap в окнах 4–6 не растёт — утечки нет.

3.  Предел по числу соединений достигнут по памяти, а не по ошибкам.

4.  Дополнительно фиксируется отношение числа сборок мусора к числу
    запросов (GC на запрос) — грубый показатель нагрузки на сборщик.
    **ВАЖНО**: величина посчтинана на всех четырёх видах нагрузки, где
    поток крупных блоков данных (длинные соединения) доминирует по
    объёму данных — это не чистая мера эффективности на один обмен, а
    ориентир по смешанному профилю.

5.  Получены служебные оценки: предельное число соединений и память на
    соединение (помечены как черновые, не для прямого расчёта capacity
    без поправок).

# Подготовка системы (prerequisites)

После перезагрузки сбрасываются `sysctl` и CPU governor — без настройки
результат может быть занижен.

    # CPU governor (должен быть performance)
    cat /sys/devices/system/cpu/cpu0/cpufreq/scaling_governor
    echo performance | sudo tee /sys/devices/system/cpu/cpu*/cpufreq/scaling_governor

    # TCP port range (должен быть 1024 65535)
    cat /proc/sys/net/ipv4/ip_local_port_range
    sudo sysctl -w net.ipv4.ip_local_port_range="1024 65535"

    # tcp_tw_reuse (должен быть 1)
    cat /proc/sys/net/ipv4/tcp_tw_reuse
    sudo sysctl -w net.ipv4.tcp_tw_reuse=1

# Запуск теста

По умолчанию `@Tag("stress")`-тесты исключены из прогона через
`surefire.excludedGroups` (корневой `pom.xml`). Переопределить можно
через `-Dsurefire.excludedGroups=`:

## Через make (рекомендуется)

    # Смешанная нагрузка, 5 минут (по умолчанию)
    make test-stress-tlssession-mixed

    # Ускоренный прогон (1 минута)
    make test-stress-tlssession-mixed ARGS=-Dstress.duration=1

    # Полный прогон для вердикта об утечке (30 минут)
    make test-stress-tlssession-mixed ARGS=-Dstress.duration=30

    # Прогон с пользовательскими параметрами
    make test-stress-tlssession-mixed \
      ARGS="-Dstress.duration=10 -Dstress.maxSessions=5000 -Dstress.step=100"

    # Только проверка предела соединений (без смешанной нагрузки)
    make test-stress-tlssession-mixed \
      ARGS="-Dtest=org.rssys.gost.tls13.stress.TlsSessionStressTest#maxConcurrentSessions"
